# optimize 详细流程

交付前给 Figma 设计稿做「开发者思维」评审并帮改到位,产一份交付就绪报告。全程**读 Figma + 经用户裁定写回 Figma**,不生成代码、不碰 figma2web。

## P0 — 前置校验
1. figma-mcp 可用(未认证 → 先跑认证流程)。
2. 向用户索取**待审的 Figma 链接 / 范围**(单页 / 若干区块)。**不落盘,每次现问。**
3. 说明本次评审依据:内置清单(见 [checklist.md](checklist.md))+ 现场抽取的项目标准。

## P1 — 装载项目标准(零 config)
1. **优先** `get_variable_defs` 从待审 Figma 抽 variables/styles → **色板 / 字体族 / 字号阶 / 间距网格基数**。
2. **抽不到** → 退回 checklist.md 的通用默认阈值,并就关键缺口(至少色板 + 网格基数)**对话补齐**。
3. **不生成、不读写任何 config 文件**,不触碰 figma2web 的 `config.md`。项目标准仅用于本次评审,不落盘。

## P2 — 逐区块提取 + 评审
对每个区块:
1. `get_metadata` + `get_screenshot` + `get_design_context` 取结构与视觉。
2. **数值类**(色 / 间距 / 圆角 / 字号)靠 `use_figma` 读 x/y/w/h/fills 精确审计,对照项目标准。
3. **视觉 / 结构类**靠截图 + design_context 判断。
4. 逐维度对照 checklist.md(A–H),产**带严重度的建议列表**,每条含:问题 / 改法 / 依据(清单编号) / 严重度。
5. 命中 snap 网格例外的合理值**不报**。

## P3 — 逐条裁定(HARD GATE)
逐区块、逐条呈交建议,用户对每条选:
- **采纳** —— 进入 P4 二次选择。
- **跳过** —— 记入报告(跳过)。
- **再调** —— 用户给反馈,AI 修正该条后重新呈交。

**写入 Figma 前必须确认,未确认不得改动任何节点。**

## P4 — 采纳项二次选择
每条采纳的问题,用户再选处置方式:
- **让 AI 改** —— 进入 P5 写回。
- **我自改** —— AI 给出明确、可照做的改法说明(改哪个节点、从什么值改到什么值、为什么),交设计师在 Figma 自行修改。

## P5 — 写回 Figma(仅「让 AI 改」项)
1. **先走 `figma-use` skill**(强制前置,调 `use_figma` 之前必做)。
2. `use_figma` 写入改动。
3. **每处改完 `get_screenshot` 校验**,确认改动符合预期。
4. 字体类改动云端受限无法执行时 → **降级为「建议」**,不强制,报告标注「受限」。

## P6 — 复审
复审**数值类**审计项是否达标:
- 理想为审计列表**归零**;
- 或仅剩用户**有意保留**、并在报告标注的例外。
未达标项回 P3 与用户确认是保留还是继续改。

## P7 — 交付就绪报告(可选落盘)
按 [report-template.md](report-template.md) 出报告:发现的问题、处置结果(已改 / 跳过 / 待改 / 受限建议)、保留的例外、给开发的交接 TL;DR。
- **默认落** `docs/figma-optimize/<页面>-<日期>.md`;
- 报告阶段**允许用户改路径或选择不落盘**。

## 完成标志
- 数值类审计项复审后归零,或仅剩报告标注的有意例外。
- 结构类、切图友好类高严重度问题均已处置(改到位或明确跳过并记录)。
- 报告产出,设计师可据它交付开发。

## 边界
- 只审既有稿并按用户裁定优化,**不无中生有设计新界面**。
- **不做审美裁决**,审美方向归设计师;AI 只揪可验证问题。
- 越出「评审 + 经确认写回 Figma + 出报告」范围即停。
