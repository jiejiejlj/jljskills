# interview2doc 访谈方法论与文档模板

梳理想法时,用「逐一穷追式访谈」把每个角落问透,再结构化成需求文档。文末的「需求文档模板」既是产出格式,也是访谈的覆盖清单——按模板章节逐一问透即可,不需要另记一份维度列表。

## 访谈方法(逐一问透)

1. **一次只问一题**,问完停下等用户反馈,再问下一题。一口气抛多题会让人懵。
2. **每题附「我的推荐答案 + 简短理由」**,不要只抛问题让用户裸答。
3. 能枚举选项的 → 用 `AskUserQuestion` 结构化提问,把推荐答案放第一个选项并标「(推荐)」。
4. 枚举不出的开放题 → 用纯文本提问,问题末尾写明推荐答案。
5. **先查能查到的**:当前项目里能读到的事实(技术栈、现有结构、约定)先自己查,别拿去问用户。
6. **按依赖排序**:先问上游问题(它决定下游有哪些选项),再逐层往下,别乱跳分支。
7. **耐心穷追**:不偷懒、不替用户跳过,把每个角落问透。但这是梳理想法,达成「想法已完整表达」即可,不必逼到对抗。
8. **警惕「全按推荐」**:连续三题用户都原样接受推荐答案时,停下来把这几题的结论串成一段整体复述,请用户确认方向。推荐答案是帮用户省力,不是替他拍板——全程照单全收写出的文档是你的想法,不是他的。
9. **矛盾即缺口**:用户的说法前后矛盾、或某条内容含糊得无法落笔(如「都行,你看着办」)时,不要静默择一、照写进文档或直接丢进开放问题——把矛盾双方摆出来、把含糊处点明,当场问回去(照旧附推荐答案)。这不是评审他的方案,而是誊清的必然要求:原稿不自洽,抄写员必须停笔问清。

为什么「每题附推荐答案」很关键:裸抛问题等于把梳理负担又甩回给用户,他本来就还没想清楚才来用这个 skill。给出带理由的推荐答案,用户只需「认同 / 修正」,既省力又能借你的判断把想法推进一步。

### 一个好问法长什么样

> **范围边界**:这个工具第一版要不要支持多人协作?
> 我的推荐:**先不支持,只做单人**(推荐)。理由:协作会牵出账号、权限、实时同步一大串复杂度,而你现在的核心痛点是「自己用着顺手」,单人版能最快验证想法。等单人跑通了再加协作也不迟。

对照反例(不要这样):「你要不要支持多人协作?」—— 没给推荐、没给理由,用户只能从零纠结。

## 需求文档模板(兼访谈覆盖清单)

访谈围绕以下章节逐一问透;汇总时按同一结构产出 Markdown。章节可裁剪但保持顺序——只裁对本需求**确实不适用**的章节;适用但没聊出结论的不裁,归入「开放问题」。各章节尖括号里的提示就是该维度要问透的内容:

```markdown
# <主题> 需求文档

> 日期:<YYYY-MM-DD> · 状态:草稿

## 1. 概述 / 背景
<一两段话讲清楚这是什么、为什么做>

## 2. 要解决的问题
<背景、痛点、现状,为什么现在做>

## 3. 目标(Goals)
<做成什么样算成功的方向>
- ...

## 4. 非目标(Non-Goals)
<明确不做什么、范围外是什么>
- ...

## 5. 目标用户与使用场景
- 用户:<谁用>
- 场景:<在什么场景下用>

## 6. 核心需求 / 功能清单
<必须有的能力,按优先级>
| 优先级 | 需求 | 说明 |
| --- | --- | --- |
| P0 | ... | ... |

## 7. 约束与依赖
<技术、时间、预算、合规、外部依赖>
- ...

## 8. 成功标准
<怎么衡量做对了,给可验证的标准>
- ...

## 9. 开放问题(待确认)
<暂时定不下来、需要后续确认的点>
- ...
```
