# pre-design — 把想法梳理成需求文档

## 用途
帮用户把一份模糊的想法 / 需求,通过耐心的逐一访谈,梳理成一份清晰、结构化的需求文档。

## 何时运行
仅当用户主动用 `/project:pre-design` 指令调用时运行。**不要由 agent 自行触发。**

## 流程

1. **读取初始输入**。用户调用时可能附带信息,按来源依次取:
   - ① 跟在指令后的描述文字;
   - ② 用户给的文档路径 / 链接 —— 本地用 Read 读、链接用 WebFetch 读进来;
   - ③ 当前项目上下文 —— 能自己查到的事实先查,别拿去问用户。

   若三者都凑不齐,第一题先问用户「这次想梳理的想法 / 需求是什么」。

2. **耐心访谈,直到想法输入完毕**。围绕「要解决什么问题、目标用户与场景、核心需求、范围边界、约束、成功标准」逐一把想法问透。
   - 访谈方法(一次一问、每题附推荐答案、按依赖排序等)见 [references/method.md](references/method.md) —— **开始提问前先读它**。
   - **每轮访谈结束,主动问一句「想法是否已经输入完毕?」**;用户未确认前,保持耐心继续问,不要急着写文档。

3. **结构化汇总**。用户确认输入完毕后,把结论填进需求文档模板(见 method.md 的「需求文档模板」)。

4. **询问落盘位置**(写文件前必做)。先问用户存到哪,给出推荐值:
   `docs/pre-design/<YYYY-MM-DD>-<主题>.md`(当前项目 docs 目录 + 本 skill 名 + 「时间-主题.md」)。

5. **写文档并请用户 review**。落盘后让用户检查,按反馈迭代修改。

## 注意
- 这是「梳理输入」不是「评审方案」:目标是把用户脑子里的东西完整、清晰地誊到文档里。
- 用户没确认输入完毕之前,不要自作主张开始写文档。
- 落盘路径一定先问,别直接写死。
