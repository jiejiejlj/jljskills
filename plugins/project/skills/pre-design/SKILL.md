# pre-design — 把想法梳理成需求文档

## 用途
用户脑子里有一个还很模糊的想法或需求,本 skill 通过耐心的逐一访谈,把它问透、誊清,产出一份结构化的需求文档,供后续设计 / 开发使用。

核心心法:**这是「梳理输入」,不是「评审方案」。** 目标是把用户脑子里的东西完整、准确地搬到纸面,而不是替他做决定或挑毛病。

## 何时运行
仅当用户主动用 `/project:pre-design` 指令调用时运行。它会主导一段访谈,不该在普通对话里被自动触发去打断用户。

## 流程

1. **读取初始输入**。用户调用时可能已经带了信息,按来源依次取,**别让用户重复已经给过的东西**:
   - ① 跟在指令后的描述文字;
   - ② 用户给的文档路径 / 链接 —— 本地用 Read 读、链接用 WebFetch 读进来;
   - ③ 当前项目上下文 —— 能自己查到的事实(技术栈、现有结构)先查。

   三者都凑不齐时,第一题就先问「这次想梳理的想法 / 需求是什么」。

2. **耐心访谈,直到用户确认输入完毕**。围绕「要解决什么问题、目标用户与场景、核心需求、范围边界、约束、成功标准」逐一把想法问透。
   - 访谈方法(一次一问、每题附推荐答案、按依赖排序)见 [references/method.md](references/method.md) —— **开始提问前先读它**,照着问能问得更透、也不会让用户疲于裸答。
   - **每轮结束主动问一句「想法是否已经输入完毕?」。** 这一步是闸门:模型容易问两三句就急着动笔,导致文档残缺。用户没点头之前,继续耐心问。

3. **结构化汇总**。用户确认完毕后,把访谈结论填进需求文档模板(见 method.md 的「需求文档模板」)。只写访谈里确认过的内容,定不下来的进「开放问题」,不要自行脑补需求。

4. **先问落盘位置,再写文件**。直接写死路径会污染用户的项目结构,所以先问、给推荐值:
   `docs/pre-design/<YYYY-MM-DD>-<主题>.md`(当前项目 docs 目录 + 本 skill 名 + 「时间-主题.md」)。

5. **写文档并请用户 review**。落盘后让用户检查,按反馈迭代修改。

## 红线(容易踩的三点)
- 用户**没确认输入完毕**之前,不要动笔写文档。
- **落盘路径一定先问**,别直接写死。
- 文档里**只放确认过的内容**,拿不准的归「开放问题」,不替用户拍板。
