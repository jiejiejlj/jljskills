# pre-design 访谈方法论与文档模板

梳理想法时,用「逐一穷追式访谈」把每个角落问透,再结构化成需求文档。

## 访谈方法(逐一问透)

1. **一次只问一题**,问完停下等用户反馈,再问下一题。一口气抛多题会让人懵。
2. **每题附「我的推荐答案 + 简短理由」**,不要只抛问题让用户裸答。
3. 能枚举选项的 → 用 `AskUserQuestion` 结构化提问,把推荐答案放第一个选项并标「(推荐)」。
4. 枚举不出的开放题 → 用纯文本提问,问题末尾写明推荐答案。
5. **先查能查到的**:当前项目里能读到的事实(技术栈、现有结构、约定)先自己查,别拿去问用户。
6. **按依赖排序**:先问上游问题(它决定下游有哪些选项),再逐层往下,别乱跳分支。
7. **耐心穷追**:不偷懒、不替用户跳过,把每个角落问透。但这是梳理想法,达成「想法已完整表达」即可,不必逼到对抗。

## 访谈要覆盖的维度

- **要解决的问题**:背景、痛点、为什么现在做。
- **目标(Goals)**:做成什么样算成功的方向。
- **非目标(Non-Goals)**:明确不做什么、范围外是什么。
- **目标用户与场景**:谁用、在什么场景下用。
- **核心需求 / 功能清单**:必须有的能力,按优先级。
- **约束与依赖**:技术、时间、预算、合规、外部依赖。
- **成功标准**:怎么衡量做对了(可验证的标准)。
- **开放问题**:暂时定不下来、需要后续确认的点。

## 收尾:确认输入完毕

每轮访谈结束,主动问「想法是否已经输入完毕?」。用户确认后,再进入文档汇总。

## 需求文档模板

汇总时按以下结构产出 Markdown(章节可按实际裁剪,但保持顺序):

```markdown
# <主题> 需求文档

> 日期:<YYYY-MM-DD> · 状态:草稿

## 1. 概述 / 背景
<一两段话讲清楚这是什么、为什么做>

## 2. 要解决的问题
<痛点、现状>

## 3. 目标(Goals)
- ...

## 4. 非目标(Non-Goals)
- ...

## 5. 目标用户与使用场景
- 用户:...
- 场景:...

## 6. 核心需求 / 功能清单
| 优先级 | 需求 | 说明 |
| --- | --- | --- |
| P0 | ... | ... |

## 7. 约束与依赖
- ...

## 8. 成功标准
- ...

## 9. 开放问题(待确认)
- ...
```
