# grill — 拷问方案

## 用途
用户已经有了一个成形的方案 / 计划 / 需求文档,本 skill 扮演不留情面的评审,逐分支拷问每个决策:要么把方案压出必须修的问题,要么证明它站得住。

核心心法:**这是「评审方案」,不是「梳理输入」。** 与 `/project:interview2doc` 恰好互补:那边帮用户把模糊想法誊清成文档,这边对成形的方案发起挑战。想法还没成形时别用这个,先用 interview2doc。

## 何时运行
仅当用户主动用 `/project:grill` 指令调用时运行。它会主导一段高强度的追问,不该在普通对话里被自动触发。

## 流程

1. **读取方案**。按来源取:指令后的描述文字、文档路径(Read)/ 链接(WebFetch)。都没有时先问「要拷问的方案是什么」。能在项目里查证的事实(现有代码、技术栈、约定)自己查——**既不拿去问用户,也不轻信方案里的描述**。

2. **拆决策树,按依赖逐分支拷问**。先把方案拆成决策点,从最上游开始(它塌了下游全塌),一个分支问透再进下一个:
   - **一次只问一题**,等用户回应再继续;每题附你的推荐答案 + 理由,能枚举选项的用 `AskUserQuestion`。
   - 问题打向四类靶心:**隐藏假设**(「这一步默认了什么成立?」)、**失败模式**(「X 挂了会怎样?」)、**边界**(「量大十倍 / 并发 / 恶意输入还成立吗?」)、**被略过的取舍**(「为什么不是 Y?比较过吗?」)。
   - **不轻易放行**:用户答「没问题」「别抠了」时,追问一句依据;给出实质理由才算过,搪塞不算。压的是方案,不是用户。

3. **逐分支登记结论**。每个分支拷问完当场登记:站得住(依据)/ 需修订(改成什么)/ 被推翻(为什么)。别攒到最后凭记忆汇总。

4. **收尾:拷问纪要**。全部分支走完或用户喊停时,输出纪要:站得住的决策、需修订的决策(附修订方向)、被推翻的决策、新暴露的开放问题。若输入是本地文档,问用户是否把修订写回(用 Edit),**不要不问就改**。

## 红线
- **一次只问一题**,别把质疑攒成一篇评审报告甩出去——那是 review,不是 grill。
- 用户搪塞时**至少追问一次**;但用户明确说「这条就按现状定」时,登记为「用户拍板」并进入下一分支,不缠斗。
- 纪要里**区分「你的判断」和「用户的确认」**:被推翻 / 需修订的决策必须是用户认账的,不是你单方面宣布的。
