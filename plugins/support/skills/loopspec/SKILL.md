# loopspec — 拷问出 workflow 规格

## 用途
把用户生活 / 工作里一个**重复的循环**,通过对抗式拷问逐决策问透,落成一份**自动化 workflow 规格**——一份实现 agent 能零提问就开工的文件。

核心心法:**「循环即可委派」。** 把工作看成一层层循环,越可预测的循环越值得交出去;一份 workflow 就是某个循环被固化成的、能直接开工的规格。

与另两个 skill 的分工(别混):
- `/support:interview2doc` 中立地把模糊需求誊成需求文档——**不挑刺、不产可执行工件**。
- `/codeflow:grill` 对抗式拷问任意已成形方案,产出拷问纪要——**只挑刺、不产工件**。
- 本 skill = **grill 的拷问手法 + 一个固定交付物(workflow 规格)**,且只在**自动化 workflow** 这一个领域;它还会用「循环透镜」帮用户**发现**值得做的循环,不只评审给定的。

## 何时运行
仅当用户主动用 `/support:loopspec` 指令调用时运行。它会主导一段高强度追问,不该在普通对话里被自动触发。

## 词汇表
一套共享语言,**按需取用,绝不当填空清单**。**结构上什么都不强制**:一个 workflow 不必有 AI、不必有检查点、不必有定时——除非拷问证明它需要。别默认往里塞。

- **触发器 Trigger**:每次运行由什么点燃。**事件**触发(来了新邮件 / 新 issue)或**定时**触发(每天早上)。事件触发通常更高效。
- **检查点 Checkpoint**:人在环的确认 / 决策点。有的 workflow 一个都没有、全自动跑;有的完全不用 AI。
- **右推 Push right**:把检查点尽量往后推。人被打扰之前尽量多做,让人只被问**一次**、在最后、且材料都备齐时。
- **简报 Brief**:检查点端给人的东西——**决策就绪**的紧凑摘要(做了什么、为什么、附一个能下钻到原始产物的链接),不是原始输出。人读的是简报,不是草稿;评审速度是硬指标。

## 工作区
工作区根目录统一 `docs/jljskills/support/loopspec/`(不存在则连同建出,位置不询问),其下:

- `workflows/*.md` —— 一个 workflow 一份规格,长期积累,是 source of truth。
- `NOTES.md` —— 记录用户的世界:他用的工具、要处理的渠道、以及他自己对这些的叫法。**空或很薄时,先访谈他的世界,再去做任何规格**。模糊的说法一出现就磨成规范词,记进这里,下次不必重问。

## 流程

1. **读 `NOTES.md`**。空 / 薄 → 先花几问访谈用户的世界(工具、渠道、术语),把确认过的记进 `NOTES.md`,再往下走。

2. **定拷问对象**。用户在指令后指定了要设计的 workflow → 直接进第 3 步;没指定 → 用「循环透镜」帮他找一个值得委派的循环,主动提出他没注意到的,选定一个再拷问。

3. **逐决策对抗式拷问**(grill 手法)。把这个循环拆成决策点,从最上游开始,一个问透再进下一个:
   - **一次只问一题**,等回应再继续;每题附你的推荐答案 + 理由,能枚举选项的用 `AskUserQuestion`。
   - 靶心 = 词汇表四问(**怎么触发?要不要检查点?检查点能不能再右推?简报里放什么?**)+ grill 四类(**隐藏假设 / 失败模式 / 边界 / 被略过的取舍**)。
   - **不轻易放行**:用户答「行了别抠了」时追问一句依据;给出实质理由才算过。压的是规格,不是用户。

4. **随拷问实时增删改规格文件**。每问透一处就写进 / 改进对应的 `workflows/<名>.md`,别攒到最后凭记忆一次落盘。冒出的模糊术语顺手磨成规范词,回写 `NOTES.md`。

## 完成判据
一份规格 done 的唯一标准:**实现 agent 能零提问地把它建出来**。只要还有一个问题悬着,就没 done,继续拷问。这比「分支走完」更硬——别问两三句就急着收工。

## 红线
- **一次只问一题**,别把拷问攒成一篇评审报告甩出去。
- **结构上什么都别默认**:没被拷问证明需要之前,不给 workflow 塞 AI / 检查点 / 定时。
- 工作区固定 `docs/jljskills/support/loopspec/`,产物不散落别处。
- `NOTES.md` 与规格里**只放确认过的内容**;拿不准的当作还没 done 的开放点继续问,不替用户脑补。
