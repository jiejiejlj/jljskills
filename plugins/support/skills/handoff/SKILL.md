# handoff — 会话交接

## 用途
把当前会话的任务现场压缩成一份交接文档,写给一个**零上下文的新 agent** 看,让它接手后不用重新摸索就能继续干活。

核心心法:**交接文档是「现场快照 + 行动指令」,不是会话流水账。** 新 agent 不需要知道对话经过了什么,只需要知道:任务是什么、走到哪了、为什么这么走、接下来干什么。

## 何时运行
仅当用户主动用 `/support:handoff` 指令调用时运行。典型时机:上下文窗口快满、今天收工明天换新会话继续、要把任务转交给另一个人或 agent。

## 流程

1. **确定交接目标**。用户在指令后带了参数(说明下一个会话要做什么)时,整份文档围绕该目标裁剪——与目标无关的细节从简;没带参数时,以「接续当前任务」为默认目标,不要为此追问用户。

2. **盘点现场**。回顾本次会话:做了什么、改了哪些文件、做过哪些决策(以及被否掉的方案)、踩过哪些坑、还剩什么没做。会话里产生过正式产物(需求文档、计划、commit、PR)的,记下路径或链接。

3. **写交接文档**。存到**系统临时目录**(如 `/tmp` 或 `$TMPDIR`),**不要写进当前工作区或仓库**——交接文档是会话间的便签,不该进 git。文件名带项目名和时间戳,如 `/tmp/handoff-<项目名>-<yyyyMMdd-HHmm>.md`。小节要求见下。

4. **交付路径**。结尾必须把文档**完整路径**明确打给用户,并附一句用法提示:在新会话里发送「请读取 <路径> 并继续其中的任务」即可接续。

## 交接文档必含小节

| 小节 | 写什么 |
| --- | --- |
| 任务与目标 | 一句话说清在做什么、做到什么程度算完 |
| 当前状态 | 已完成 / 进行中 / 未开始,逐条列 |
| 关键决策与理由 | 定了什么、为什么,包括被否掉的备选方案(防止新 agent 重新提起) |
| 踩过的坑 | 走过的死胡同、试错结论,防止新 agent 重走 |
| 相关产物索引 | 涉及的文件、文档、commit、PR、issue,**只写路径或链接** |
| 下一步行动 | 按优先级排序的具体步骤,第一条要细到能直接动手 |
| 建议调用的 skills | 推荐新会话应调用哪些 skill、在哪一步调用(如先 `/project:grill` 拷问方案再动手) |

## 红线
- **已有产物只引用不复制**:需求文档、计划、ADR、issue、commit、diff 里已有的内容,一律写路径 / URL,不把内容抄进交接文档。
- **脱敏**:API key、密码、token、个人身份信息等敏感内容一律替换为 `[REDACTED]`,不进文档。
- **不污染工作区**:文档只写到系统临时目录;用户明确指定其他位置时才例外。
- **写给零上下文读者**:不出现「如前所述」「刚才那个方案」这类依赖会话记忆的指代,每个名词第一次出现就说清是什么。
