# 意图文档释义 — ADR / PRD / CONTEXT.md / DESIGN.md / PRODUCT.md

本 skill（及 `improve-arch`）在侦察阶段都要求先读这几类文档，原文统称 **intent docs（意图文档）**。它们记录的是**代码本身说不出来的东西**：为什么这么做、给谁做、用什么词说话。不读它们，审计者会把「深思熟虑的取舍」误报成「问题」，把「已被拒绝的方向」当新建议重提。

## ADR — Architecture Decision Record（架构决策记录）

一份短文档记**一个**架构决策，编号存放（如 `docs/adr/0001-xxx.md`），标准结构三段：**背景**（当时面临什么问题）、**决策**（选了什么、放弃了什么）、**后果**（接受了什么代价）。

解决的痛点：决策的*理由*会从代码里蒸发。半年后没人记得「为什么这里用同步写」，新人或 AI 审计者就会把它当 bug 重新提出来——ADR 是防止已定决策被反复重新争论的档案。

对审计的意义：**ADR 记录的取舍是 by-design，不是 finding**；但**过期的 ADR 本身是 finding**——代码已漂移离文档所记决策时，要上报「决策漂移」（文档或代码总有一个错了），而不是拿文档压掉发现。

## PRD — Product Requirements Document（产品需求文档）

回答「**做什么、为谁做、为什么现在做**」：目标用户、使用场景、功能范围（含明确不做什么）、成功标准。管产品意图，不管实现方式。

对审计的意义：方向类（direction）审计的最强证据来源——「PRD 承诺但代码没跟上」是最扎实的方向建议依据；PRD 已明确拒绝的方向，审计者不许再提（发现矛盾时上报矛盾本身）。

## CONTEXT.md — 领域词汇表（非缩写，即「上下文」）

一份**共享领域词汇表**：本项目的关键名词各叫什么、精确指什么。思想源头是 DDD（领域驱动设计）的「统一语言」——团队、代码、文档全用同一套词，翻译成本归零。

对审计的意义：finding 与计划里的领域名词一律用它的——执行者（人或模型）不必猜「他说的 service 是不是我们的 module」。

## DESIGN.md — 设计系统规范

前端 / UI 项目的**设计系统说明**：设计 token（色板、间距、字号）、组件清单、视觉约定。

对审计的意义：（1）遵守设计系统而产生的重复样式不是技术债；（2）UI 相关计划要指定「复用这些 token / 组件」，执行者不会自造一套颜色。

## PRODUCT.md — 产品简介

一页纸的**常驻产品简报**：这项目是什么、给谁用、核心价值。与 PRD 的区别：PRD 面向「一次要做的事」，PRODUCT.md 是长期活文档。

## 五者一张表

| 文档 | 记录什么 | 回答的问题 | codeflow 里的生产者 |
|---|---|---|---|
| ADR | 已定的架构取舍 | 「为什么当初这么选」 | `build-context` 落笔 `adr/` |
| PRD | 一次交付的需求 | 「这次做什么、为谁」 | `to-prd` |
| CONTEXT.md | 领域词汇 | 「这个词在本项目精确指什么」 | `build-context` |
| DESIGN.md | 设计系统 | 「UI 该长什么样、用什么 token」 | （暂无） |
| PRODUCT.md | 产品定位 | 「这项目是什么、给谁用」 | （暂无） |

注意咬合关系：**codeflow 自己生产其中三样（ADR、PRD、CONTEXT.md），本 skill 是它们的消费者**——在跑过 codeflow 交付链的项目上审计，意图文档天然齐备。
