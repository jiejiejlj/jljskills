# diagnosing-bugs — 反馈回路优先

硬 bug 与性能回归的诊断纪律。跳过任何阶段须显式说明理由。探索代码库时读 `docs/jljskills/codeflow/CONTEXT.md` 与 `docs/jljskills/codeflow/adr/`（不存在则跳过）。

## Phase 1 — 建反馈回路

**这就是本 skill 的全部，其余都是机械活。** 有一条能对本 bug 变红的信号，你就能找到病因——二分、假设检验、插桩都只是消耗它；没有它，读多少代码都救不了你。在这一步花不成比例的精力：激进一点，有创造力一点，别轻易放弃。

构造手段见 [references/loop-recipes.md](references/loop-recipes.md)。

### 完成判据

Phase 1 完成于：你能指出**一条命令**——脚本路径、测试调用、curl 之类——**已经跑过至少一次**（贴出调用与输出），且满足四条：

- [ ] **红能力**——驱动真实 bug 代码路径，断言**用户的确切症状**；不是「没报错」，必须真能在这个 bug 上变红、修好后变绿。
- [ ] **确定性**——同一判定每次复现；flaky bug 用钉住的高复现率替代。
- [ ] **快**——秒级，不是分钟级。
- [ ] **可无人值守**——能不看着跑；人在环仅经 [scripts/hitl-loop.sh](scripts/hitl-loop.sh) 结构化驱动。

### 硬门

**没有能变红的命令，就没有 Phase 2。** 先建回路，再谈假设——读代码攒理论，正是本 skill 要防的那个失败。抓到自己在这一步之前就开始「读代码攒理论」，立刻停下。

### 逃生通道

实在建不出回路：明说，列出已经尝试过什么，向用户要三样之一：(a) 能复现问题的环境访问权限，(b) 捕获的工件（HAR / 日志 / core dump / 带时间戳的录屏），(c) 临时生产插桩的许可。没有回路，绝不进入假设阶段。

## Phase 2 — 复现 + 最小化

跑回路，看它变红——bug 出现。确认三点：

- [ ] 复现的是**用户描述的那个失败**，不是隔壁一个恰好也在报错的失败——认错 bug 等于修错地方。
- [ ] 失败可重复（非确定性 bug：达到足以调试的复现率）。
- [ ] 已捕获确切症状（错误信息 / 错误输出 / 耗时），供后续阶段验证修复是否真的对症。

### 最小化

变红之后，把复现缩到**仍能变红的最小场景**：逐项裁——输入、调用方、配置、数据、步骤——**一次裁一项**，每裁一刀重跑一次回路，只留下承重的部分。

不复现、不最小化，不许进 Phase 3。完成判据：剩下每一项都承重——去掉任意一项，回路就会变绿。

## Phase 3 — 假设

先列 3–5 个排好序的假设，再动手验证任何一个——单假设直奔第一个可信念头会锚定判断。

每条假设必须**可证伪**：

> 格式：「若 X 是因，则改 Y 会让 bug 消失 / 改 Z 会让它更糟。」

说不出预测的假设是玄学——弃掉或磨尖。

**列表先给用户看**（他们常有能瞬间重排序的领域知识，或者已经排除过的假设——低成本检查点，省下大量时间）。用户不在场不阻塞，按自己的排序继续。

## Phase 4 — 插桩

每个探针对应 Phase 3 里一条具体预测，一次只改一个变量。

工具优先级：

1. 调试器 / REPL 检查（环境支持时）——一个断点顶十条日志。
2. 定向日志，打在能区分假设的边界上。
3. 绝不「全打出来再 grep」。

每条调试日志带**唯一前缀**，如 `[DEBUG-a4f2]`——收尾时一次 grep 清干净；不带前缀的日志会赖着不走。

性能分支：日志此时通常没用。先建立基线测量（计时 / `performance.now()` / 剖析器 / 查询计划），再二分。**先测量，后修。**

## Phase 5 — 修复 + 回归测试

修复前先写回归测试——但**仅当存在正确 seam 时**才写（seam 判断 Read [`../design-deep-module/SKILL.md`](../design-deep-module/SKILL.md)）。

正确 seam：测试在调用现场按真实 bug 模式行使——不是只有单一调用方的浅测试却要覆盖多调用方触发的 bug。

**没有正确 seam，这件事本身就是发现**——记录下来，标给 Phase 6 复盘，不要在错的 seam 上硬凑一个测试换取假信心。

有正确 seam 则：

1. 把最小化复现变成该 seam 上的一条失败测试。
2. 看它红。
3. 修复。
4. 看它绿。
5. 回到原始（未最小化）场景，重跑 Phase 1 的回路。

## Phase 6 — 清理 + 复盘

宣布完成前，五条 checkbox：

- [ ] 原始复现已不再复现（重跑 Phase 1 回路验证）。
- [ ] 回归测试已通过（或已记录无正确 seam）。
- [ ] 所有 `[DEBUG-...]` 插桩已清（grep 前缀确认清空）。
- [ ] 一次性 harness / 原型已删净，或移到明确标注的 debug 位置。
- [ ] 验证成立的假设写进了 commit / PR 信息，供下一个调试者受益。

**然后问：什么本可以防住这个 bug？** 答案若是架构性的（没有好 seam、调用方纠缠、隐藏耦合），带着具体细节指路 `/codeflow:improve-arch`。这个建议要在修复落地**之后**提，不是之前——现在你手上的信息比开工时多得多。
