# 回路构造法

供 [`../SKILL.md`](../SKILL.md) Phase 1 用：如何造出一条能对本 bug 变红的反馈回路。建对回路，bug 就修完了九成。

## 十种构造法——大致按此顺序尝试

1. **失败测试**，放在任何能到达 bug 的 seam 上——unit、integration、e2e 都行。
2. **curl / HTTP 脚本**，打向跑起来的 dev server。
3. **CLI 调用**，喂一个 fixture 输入，diff stdout 对比已知正确的快照。
4. **无头浏览器脚本**（Playwright / Puppeteer）——驱动 UI，断言 DOM / console / network。
5. **回放捕获的 trace**：把真实网络请求 / payload / 事件日志存到磁盘，隔离重放通过这条代码路径。
6. **抛弃式 harness**：拉起系统的最小子集（一个服务、mock 掉依赖），用单个函数调用触达 bug 代码路径。
7. **property / fuzz 回路**：如果 bug 是「偶尔输出错」，跑 1000 个随机输入找失败模式。
8. **bisect harness**：如果 bug 出现在两个已知状态之间（commit、数据集、版本），自动化「置状态 X → 检查 → 重复」，做到可以 `git bisect run`。
9. **差分回路**：同一输入分别跑旧版本 vs 新版本（或两种配置），diff 输出。
10. **HITL bash 脚本**（最后手段）：必须由人来点击时，用 [`../scripts/hitl-loop.sh`](../scripts/hitl-loop.sh) 结构化驱动这个人，回路依然是回路，捕获输出回流给 agent。

## 收紧回路

有了*一条*回路之后，把它当产品来打磨，问三句：

- **能更快吗？**（缓存 setup、跳过无关初始化、缩小测试范围。）
- **信号能更锐吗？**（断言具体症状，不是「没崩」。）
- **能更确定吗？**（钉住时间、seed RNG、隔离文件系统、冻结网络。）

30 秒的 flaky 回路约等于没有回路；2 秒的确定性回路是超能力。

## 非确定性 bug

目标不是一次干净复现，而是**把复现率拉高**：把触发条件循环 100 次、并行跑、加压、缩窄时间窗、注入 sleep。50% 复现率可调试；1% 不可调试——先把率拉上去，再进 Phase 2。
