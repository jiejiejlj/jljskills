# PRD 模板

七节顺序固定，逐节写：

## 问题陈述

用户视角描述用户面对的问题。只说问题，不夹带方案。

## 方案

用户视角描述这个问题打算怎么解决。仍然是用户能看懂的语言，不下沉到实现层。

## 用户故事

LONG 编号清单，覆盖尽量全——把功能涉及的角色、路径、边界情形都列到，宁多勿少。每条格式：

```
1. 作为<角色>，我想要<功能>，以便<收益>
```

示例：

```
1. 作为移动银行客户，我想要看到账户余额，以便对自己的消费做出更明智的判断
```

## 实现决策

已敲定的实现层决策，包括：

- 将新建/修改的 module 及其 interface 变更
- 开发者给出的技术澄清
- 架构决策
- schema 变更
- API 契约
- 具体的交互方式

不写具体文件路径与代码片段——它们会很快过期。唯一例外：原型产出的片段比散文更精确地固化了一个决策（状态机、reducer、schema、类型形状），可以内联在对应决策旁，注明来自原型、裁到决策相关部分。

## 测试决策

已敲定的测试层决策，包括：

- 好测试的标准：只测外部行为，不测实现细节
- 哪些 module 会被测试
- 仓库内同类测试的既有先例

## 范围外

明确写出这份 PRD 不打算做的事，划清边界，防止范围蔓延。

## 附注

其他值得记录但不属于以上任何一节的信息。
