# UI 原型

在一个路由上生成**几个长相差得很大的 UI 变体**，靠一条浮在底部的切换条来回切。用户在浏览器里翻着看，选一个（或从每个里拿点自己喜欢的），剩下的全扔。

问题是关于逻辑/状态而不是长相——分支选错了，回 [logic.md](logic.md)。

## 什么时候该走这条分支

- 「这个页面该长什么样」
- 「这个仪表盘我想先看几个方案再定」
- 「设置页面换个布局试试」
- 任何用户原本要花一天在脑子里对比三个模糊方案的场合。

## 两个子形态——强烈偏向子形态 A

UI 原型只有**贴着真实应用**才好判断——真 header、真 sidebar、真数据、真密度。一个孤零零的原型路由是真空环境：单独看每个变体都顺眼。只要有一个说得过去的既有页面能挂靠，就默认走子形态 A；原型确实没有邻近的家，才用子形态 B。

### 子形态 A——改造既有页面（首选）

路由已经存在。变体渲染在**同一个路由**上，用 `?variant=` 这个 URL search param 控制。既有的数据获取、参数、鉴权都不动——只换渲染出来的那部分。除非有明确理由不这么做，一律先选这个。

要原型化的东西还没有页面，但**天然该活在某个页面里**（仪表盘的新板块、设置页的新卡片、既有流程的新一步）——这仍然算子形态 A。把变体挂进宿主页面里。

### 子形态 B——新建页面（最后手段）

只有要原型化的东西确实没有任何既有页面能挂靠时才用——比如一个全新的顶层界面，或一个塞不进任何现有位置的流程。

按项目既有路由约定建一个**临时路由**——别为它另起一套顶层结构。命名让人一看就知道是原型（比如路径或文件名里带 `prototype` 字样）。同样用 `?variant=` 这套。

选子形态 B 之前先自问：真的没有既有页面能塞进去吗？一个空路由会藏起本该在真实场景里暴露的设计问题。

两种子形态用的浮动切换条完全一样。

## 做法

### 1. 写下问题，定 N

默认 **3 个变体**。超过 5 个就不再是「长相差得很大」，变成噪音——到 5 就该封顶。

在原型所在位置或文件顶部注释里写一行计划：

> 「设置页三个变体，用 `?variant=` 切换，挂在既有的 `/settings` 路由上。」

用户在不在场都用得上这一行。

### 2. 生成长相差得很大的变体

逐个起草每个变体，都对照这几条：

- 页面本身的目的，以及它能拿到的数据。
- 项目的组件库/样式体系（TailwindCSS、shadcn、MUI、纯 CSS，随项目而定）。
- 一个清楚的导出组件名，比如 `VariantA`、`VariantB`、`VariantC`。

变体必须**结构上不同**——不同布局、不同信息层级、不同的主要交互入口，不是只换个颜色。三个稍微调了调的卡片网格不算 UI 原型，那是壁纸。两份草稿长得太像，重做一份，明确要求「不要用卡片网格」之类的约束。

### 3. 把变体接起来

在路由上建一个单一的切换组件：

```tsx
// 伪代码 —— 按项目框架自行调整
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

子形态 A（既有页面）：所有既有数据获取逻辑留在切换组件之上，只换渲染出来的子树。

子形态 B（新页面）：`/prototype/<name>` 下的临时路由挂同一个切换组件。

### 4. 搭浮动切换条

一个固定在屏幕底部居中的小条，三块内容：

- **左箭头**——切到上一个变体（首尾相连）。
- **变体标签**——显示当前变体的 key，变体如果导出了名字，也一并显示。例如 `B — 侧边栏布局`。
- **右箭头**——切到下一个（首尾相连）。

行为：

- 点箭头更新 URL search param（用框架自带的路由方法——Next 的 `router.replace`、React Router 的 `navigate` 等），让变体可分享、可刷新保持。
- 键盘 `←`、`→` 也能切换。焦点在 `<input>`、`<textarea>`、`[contenteditable]` 上时，别拦截方向键。
- 视觉上明显跟页面本身区分开（比如高对比度的胶囊形状、淡淡的阴影），让人一看就知道这不是要被评估的设计的一部分。
- 生产构建里隐藏——用 `process.env.NODE_ENV !== 'production'` 或等价判断挡住，避免原型代码不小心合并后把切换条带给真实用户看到。

切换条做成一个共享组件，两种子形态都复用它。放在项目里共享 UI 组件通常放的位置。

### 5. 交给用户

把 URL（以及 `?variant=` 的取值）发给用户，他们有空就自己切着看。最有价值的反馈往往是「我想要 B 的 header 配上 C 的侧边栏」——那才是他们真正想要的设计。

### 6. 记下答案再清理

某个变体胜出后，把选了哪个、为什么写下来（commit message、ADR、issue，或者用户还没来得及回应时先写在原型旁的 `NOTES.md` 里）。然后：

- **子形态 A**——删掉落选的变体和切换组件，把赢家的实现并进既有页面。
- **子形态 B**——把赢家的变体提升成正式路由，删掉临时路由和切换组件。

别把变体组件或切换条留在仓库里烂掉，它们会很快过期，还会让下一个读代码的人糊涂。

## 反模式

- **变体只在颜色或文案上不同。** 那是微调，不是原型。真正的变体在结构上意见不合。
- **变体之间共享太多代码。** 共享一个 `<Header>` 没问题；共享一个 `<Layout>` 就毁了原型的意义。每个变体都该能自由推翻布局。
- **给变体接真实的写操作。** 只读的原型没问题；变体需要写操作时接一个 stub——问题是「这东西该长什么样」，不是「后端能不能跑通」。
- **把原型直接提升成生产代码。** 变体代码是在原型的约束下写的（无测试、错误处理最简），并入正式代码前要重新好好写一遍。
