# triage — 状态机分诊

把外来 issue（以及 issue-tracker.md 声明开启 PR 请求面时的外部 PR）推过一套小状态机：归类、验证、必要时拷问，最终定下一步该做什么。

## 1. 角色与状态机

两个**类别**角色：`bug`（坏了）、`enhancement`（新功能/改进）。

五个**状态**角色：`needs-triage`（待评估）、`needs-info`（等报告人补充）、`ready-for-agent`（规格齐备，可交给 agent）、`ready-for-human`（需要人来做）、`wontfix`（不予处理）。

标签的具体名字、打/摘标签与评论/关闭的命令一律照 `docs/jljskills/codeflow/issue-tracker.md`——本 skill 不复述。追踪器未配置（目标项目无 `docs/jljskills/codeflow/issue-tracker.md`）时，先指路 `/codeflow:config` 配好再回来——triage 每一步都要读写追踪器，缺配置无法开工。

不变量：每条被 triage 的 issue **恰好挂一个类别 + 一个状态**。两个状态角色同时挂着即冲突，先停下问维护者，不要自作主张摘掉一个。

转移规则：未打标签的 issue 默认先进 `needs-triage`；从这里转到 `needs-info`、`ready-for-agent`、`ready-for-human`、`wontfix` 之一；`needs-info` 一旦报告人回复，回到 `needs-triage` 重新评估。维护者可以在任何时候直接覆盖状态；但看起来异常的转移（比如 `wontfix` 突然改回 `ready-for-agent` 却没有新信息）要先问一句再动手，不要默认它是对的。

PR 面（仅当 issue-tracker.md 声明 GitHub 且 PR 请求面「开」时才存在）：**一个 PR 就是带代码的 issue**——同一套角色、同一套状态、同一台状态机。区别只在状态怎么读：`ready-for-agent` 对 PR 的意思是简报已经贴在评论区，agent 该在现有 diff 上做下一步（补测试、堵缺口），不是从零建；`ready-for-human` 对 PR 就是等人来合并。

## 2. AI 声明

triage 期间发到追踪器的每一条评论或 issue，开头必须是这一句英文原句（面向公共受众，不翻译）：

> *This was generated by AI during triage.*

## 3. 调用方式

维护者用自然语言描述意图，直接理解并执行，不强制固定指令格式。四类典型示例：

- "看看有什么要办的"
- "看看 #42"
- "把 #42 挪到 ready-for-agent"
- "有什么 agent 能领的活"

## 4. 呈现待办

查追踪器，按最旧优先呈现三个桶：

1. **未打标签**——从没 triage 过。
2. **`needs-triage`**——评估中。
3. **`needs-info` 且报告人有新动静**（自上次 triage 笔记之后有回复）——需要重新评估。

PR 面开启时，把外部 PR 并入这三个桶，每行标 `[PR]` 或 `[issue]`。discovery（呈现待办这一步）只捞**外部** PR——协作者在途的 PR 不算 triage 活；但这条过滤只管 discovery，维护者点名一个具体 PR 时，无论作者是不是协作者都照办。

每个桶给计数和逐条一行摘要，交给维护者挑选，不代选。

## 5. 办一个 issue/PR：五步

1. **gather（取材）**。读全文（正文、评论、标签、作者、时间；PR 还要读 diff）。有旧的 triage 笔记先读，报告人已经答过的问题不重问。按项目领域词汇探索代码库，尊重该区域已有的 ADR。两项排查：(a) **冗余检查**——按领域概念（不是原话措辞）搜代码库里是否已有实现，报告搜过哪些地方；命中了直接算「已实现」的 `wontfix`（进第 5 步）。(b) **先例检查**——读 `docs/jljskills/codeflow/out-of-scope/` 下全部文件，按概念相似（不是关键词）匹配有没有跟这条类似的历史否决，命中就在下一步提出来。
2. **recommend（给建议）**。把你的类别 + 状态推荐、理由，连同一份跟这条相关的代码库摘要（包括是否已实现）告诉维护者，等指示，不要自己拍板往下走。
3. **verify（验证）**。拷问之前先核实这条本身站不站得住：bug 按报告人给的步骤复现；PR 就 checkout 下来，跑相关测试/命令确认 diff 真做到了它声称的事。三种结果之一：确认（附代码路径）、复现失败、细节不足以判断（这是很强的 `needs-info` 信号）。验证通过的 issue，写出来的 agent 简报会明显更扎实。
4. **grill（拷问，按需）**。规格不够扎实需要补细节时，指路 [`../grill-with-docs/SKILL.md`](../grill-with-docs/SKILL.md)——逐题拷问、领域术语和决策当场落笔，本 skill 不重复它的流程。
5. **apply（落地结果）**：
   - `ready-for-agent`——按 [references/agent-brief.md](references/agent-brief.md) 的写法贴一条 agent 简报评论。
   - `ready-for-human`——结构跟 agent 简报一样，但要写明为什么不能交给 agent（需要判断力、外部权限、设计决策、人工测试等）。
   - `needs-info`——贴 triage 笔记（模板见第 7 节）。
   - `wontfix`——关闭，评论内容按原因三分：
     - **已实现**——指出实现在哪，**不写入** out-of-scope（那个知识库只记「被否决」的请求，不记「已经做了」的）。
     - **bug 被否决**——礼貌说明理由后关闭。
     - **enhancement 被否决**——按 [references/out-of-scope.md](references/out-of-scope.md) 写入知识库、评论里链接过去，再关闭。
   - `needs-triage`——原地打这个状态标签；有部分进展可以顺手留一条评论，非必须。

## 6. 快速覆盖与续接

维护者点名把某条直接挪到某个状态（比如「把 #42 挪到 ready-for-agent」）时，直接信任这个指示：确认一句要做的动作（改标签、发评论、关闭）之后立刻执行，跳过第 5 步的拷问。挪到 `ready-for-agent` 且没跑过拷问时，多问一句要不要顺手写一份 agent 简报。

处理一条之前，先看有没有旧的 triage 笔记；有就读一遍，看报告人是不是已经答了里面挂着的问题，答过的不重问，直接在这些答案基础上继续。

## 7. needs-info 笔记模板

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

拷问期间敲定的每一点都要收进「已确认的」，别让它随对话流失。「还需要你提供的」里每个问题必须具体、答得上来——不能是「麻烦补充更多信息」这种空话。

## 8. 边界

`/codeflow:to-issues` 产出的 issue 在发布时已经是 agent-ready，不要拿它们跑 triage。
