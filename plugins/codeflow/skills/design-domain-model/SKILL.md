# design-domain-model — 领域建模思想

## 用途

领域语言的判据单一真相源——**什么算术语坏了、什么决策值得记档**。本 skill 是纯思想基石：只供 Read 装载，不操作任何文件；发现病灶后手怎么动、写到哪，归 `../build-context/SKILL.md`。单独 `/codeflow:design-domain-model` 只有一个场景——想在普通讨论里装载这套判据。

（理论出处：DDD 的通用语言 / ubiquitous language。）

## 通用语言三原则

- **一词一义**：一个术语在一个 context 内只指一个概念，一个概念只有一个规范名。
- **单一权威**：词汇表（CONTEXT.md）是术语的唯一真相源——讨论、文档、代码命名都从它取词。
- **词与代码一致**：词汇表说的和代码做的必须互相印证；不一致就是缺陷，二者必改其一。

## 三病灶识别

- **冲突**：用词与词汇表既有定义相抵——同一个词被赋了第二个意思。
- **超载**：一个词背着多个概念（account 既指 Customer 又指 User）——要拆成多个规范词。
- **模糊**：说法宽泛到判不出边界（「处理一下」「相关数据」）——要磨成能进词汇表的精确词。

## 四手法判断内核（何时用哪招）

- **挑战**：用词与词汇表冲突时——当场对质，不等收尾。
- **锐化**：超载或模糊词出现时——提议精确规范词，给候选让用户选。
- **边界场景压测**：概念之间的分界含糊时——发明压边界的具体场景，逼出精确分界。
- **代码对质**：用户陈述与代码可能不一致时——先查代码再摆矛盾，不凭印象。

## ADR 三条件门槛（何时值得记档）

1. **难逆**——之后反悔的成本是实打实的。
2. **无上下文会困惑**——未来读代码的人会想「当年为什么这么干？」
3. **真实取舍**——存在真的备选项，且是为具体理由选的这个。

缺任何一条就跳过——ADR 是稀缺记录，不是会议纪要。

## 判据红线

- **词汇表只是词汇表**：不装实现细节、不当 spec、不当草稿本——混入实现细节的词条本身就是病灶，不是沉淀。
