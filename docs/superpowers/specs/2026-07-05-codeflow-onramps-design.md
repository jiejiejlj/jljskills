# codeflow 汇入层三件设计：triage / diagnosing-bugs / research

- **diagnosing-bugs 沿用原名**、**照家规 disable-model-invocation**（否决 model-invoked 例外与 diagnose 改名提议）。
- 约定推导项（沿既有约定，无需再议）：三件均入 codeflow；`.out-of-scope/` → `docs/jljskills/codeflow/out-of-scope/`；research 产物 → `docs/jljskills/codeflow/research/`；triage 标签固定五规范名不做映射层。

## 1. 布局（codeflow 13 → 16 skill）

```
plugins/codeflow/skills/
├─ triage/            SKILL.md + references/agent-brief.md + references/out-of-scope.md
├─ diagnosing-bugs/   SKILL.md + references/loop-recipes.md + scripts/hitl-loop.sh
└─ research/          SKILL.md（保持原版极简，十来行）
```

三件均 `disable-model-invocation: true`。README 分层四层扩五层：新增**汇入层**（triage、diagnosing-bugs），research 归入口层旁路。

## 2. triage

- **状态机保真**：2 类别角色（bug/enhancement）× 5 状态角色（needs-triage / needs-info / ready-for-agent / ready-for-human / wontfix），含全部转移规则、「恰好一类别一状态」不变量、冲突时先问维护者。
- **PR 请求面**：仅 GitHub 形态且 issue-tracker.md 声明开启时生效。discovery 只捞外部 PR（协作者在途 PR 非 triage 工作），点名 PR 一律办；PR 验证=checkout 跑相关测试确认 diff 属实；ready-for-agent 对 PR 的语义=brief 已附、agent 对既有 diff 做下一步。
- **五步流程保真**：gather（读全文+先前 triage 笔记；冗余检查按领域概念搜既有实现并报告搜索位置；先例检查读 out-of-scope/ 全部文件按概念相似匹配）→ recommend（类别+状态建议+理由，等维护者指示）→ verify（bug 按报告步骤复现；PR 验 diff；结果三态：确认/失败/细节不足即强 needs-info 信号）→ grill（需要时指路 `../grill-with-docs/SKILL.md`——它本身就是「拷问+术语当场落笔」的组合入口，正对应原版 grilling+domain-modeling 联动）→ apply（五状态各自动作；wontfix 三分支：已实现→指路所在处**不入库**防污染 / bug 拒绝→礼貌解释关闭 / enhancement 拒绝→写 out-of-scope/ 并链接后关闭）。
- **快速状态覆盖**：维护者点名「把 #42 挪到 X」时信任直达，确认动作后执行、跳过拷问；挪 ready-for-agent 时问要不要写 brief。**续接会话**：读旧 triage 笔记，不重问已解决的问题。**needs-info 笔记模板**（已确认的/还需要的两段式）保留。
- **AI 声明语**：每条 triage 评论开头保留英文原句 `> *This was generated by AI during triage.*`（GitHub 受众常含国际用户），SKILL.md 用中文说明此规则。
- **references/agent-brief.md**：耐久性四原则（不写文件路径行号、行为而非过程、可独立验证的验收标准、显式范围外）+ 模板 + bug/enhancement/PR 三个好例一个坏例，本地化搬。
- **references/out-of-scope.md**：一概念一文件、命名规范、durable 理由写法、何时查（gather 阶段全读、按概念非关键词匹配、三种维护者回应）、何时写（仅被拒 enhancement；已实现不入库）、反悔即删文件不翻旧案，全套本地化搬。目录钉 `docs/jljskills/codeflow/out-of-scope/`。
- **标签**：固定五规范名 + bug/enhancement，不做映射层（原版 setup 的 triage-labels 节裁掉）。

## 3. config 与 tracker 模板扩展（triage 的前置）

- **config SKILL.md**：选 GitHub 后恢复原版追问「外部 PR 是否请求面？」（先解释后问，默认否），答案记入目标项目 issue-tracker.md；本地 markdown 形态无此问。
- **tracker-github.md**：补 triage 操作节——五状态+两类别标签的规范名与 gh 命令（label 增删、评论、关闭）、PR 面开启时的 PR 查询与外部作者判断。
- **tracker-local.md**：frontmatter 扩展 `category:`（bug/enhancement）与 `state:`（五规范值，均可选——to-issues 产的 issue 无需这两字段）；双写规则钉死：`state: ready-for-agent` 时同步写 `agent_ready: true`（to-issues/implement 既有消费字段不变）。「to-issues 产出的 issue 已 agent-ready、不要 triage」的边界写入 triage SKILL.md。

## 4. diagnosing-bugs

- **六阶段骨架全保真**，Phase 1 灵魂门槛留正文：完成判据四条 checkbox（能对本 bug 变红且断言用户的确切症状 / 确定性或钉住的高复现率 / 秒级 / 可无人值守）+「没有变红命令禁止进 Phase 2/3，先建回路」硬门 +「实在建不了要明说、列出尝试、向用户要环境/工件/插桩许可」逃生通道。
- **references/loop-recipes.md**（下沉菜单性内容）：10 种回路构造法（失败测试→curl→CLI→无头浏览器→trace 回放→抛弃式 harness→property/fuzz→bisect 挂钩→差分回路→HITL 兜底）+ 收紧回路三问（更快/信号更锐/更确定）+ 非确定性 bug 提复现率手法。
- **后续阶段保真**：Phase 2 复现+最小化（逐项裁、每项都承重才算完）；Phase 3 一次 3–5 个可证伪假设防锚定、格式「若 X 是因，改 Y 应消失/改 Z 应更糟」、列表先给用户看但不阻塞；Phase 4 探针一次一变量、调试器优先、日志带唯一前缀 `[DEBUG-xxxx]`、性能分支先测量后修；Phase 5 修复前先写回归测试**但仅在正确 seam 存在时**——无正确 seam 本身就是发现，记录并移交；Phase 6 清理清单五条 + 复盘问「什么本可防住」，架构性答案指路 `/codeflow:improve-arch`。
- **scripts/hitl-loop.sh**：模板搬入（先例：git-policy/scripts/），step/capture 两个 helper 逻辑不动，注释与提示文案中文化。
- **衔接改造**：探索时读 `docs/jljskills/codeflow/CONTEXT.md` 与 adr/（不存在则跳过）；seam 判断 Read `../design-rules/SKILL.md`。
- description 写全触发场景词（诊断/debug/坏了/报错/间歇性失败/变慢）+「仅当用户主动用 `/codeflow:diagnosing-bugs` 指令调用时使用」。

## 5. research

保持原版极简（十来行）。三步骨架：后台代理（Agent 工具 `run_in_background: true`）→ 只查一手来源（官方文档/源码/规范，每条结论溯到拥有它的源头）→ 落成单个 Markdown 逐条带引用。改造两点：产物路径钉 `docs/jljskills/codeflow/research/<slug>.md`；结尾一句衔接——产物可作 `/codeflow:grill-with-docs` 的输入，research 喂养思考不替代思考。

## 6. 登记面

1. codeflow README：五层结构（地基/入口/交付/健康/汇入）；「怎么选」补三条（外来 issue/PR 堆积→triage；东西坏了难查→diagnosing-bugs；要查资料外包阅读→research）；文档根清单补 out-of-scope/、research/。
2. 仓库 README 分类表 codeflow 行：13→16 skill 导览更新（目录树同步）。
3. marketplace.json / CLAUDE.md：无需改动。
4. 三处 name 一致、references 相对路径纪律（../../ 上跳两级）、注释中文化等家规照旧；全程本地 commit、终审后单次 push。

## 7. 范围外

- 原版 ask-matt（路由由 codeflow README 承担）、setup 其余部分、engineering 目录外技能（teach、writing-great-skills 等）不搬。
- 不为 GitLab 加 PR 面；不做标签映射层；不自动匹配 out-of-scope（人确认匹配，原版如此）。

## 8. 风险与对策

- **triage 正文规模**：完整搬 PR 面后正文偏长——用五步流程为骨架、brief 与 out-of-scope 全量下沉 references 控制；评审按重复沉积纪律把关。
- **本地 tracker 字段演进**：state 与 agent_ready 双写规则单点钉在 tracker-local.md，防两处真相。
- **config 回填**：已发版的 config 加追问属行为变更，无破坏性（旧 issue-tracker.md 无 PR 声明=视为关闭）。
