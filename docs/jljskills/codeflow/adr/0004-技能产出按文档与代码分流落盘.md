# 技能产出按文档与代码分流落盘

技能在目标项目里的产出按**消费者**分三类落盘，判据写死在各 SKILL.md、不再向用户询问位置：

- **文档**（人读的沉淀物——需求文档、workflow 规格、词汇表、ADR、interface 草图、调研、评审报告、out-of-scope 知识库）：一律落 `docs/jljskills/<plugin>/` 根下，按 skill 或用途分子目录，目录不存在则惰性创建。
- **代码与配置**（目标项目工具链消费——源码、组件、registry、`.claude/settings.json`、hook 脚本、CLAUDE.md 段落）：按目标项目自身结构落盘，绝不进文档根。
- **临时物**（一次性呈现物如 improve-arch 的 HTML 报告、handoff 的交接便签）：写 OS 临时目录，不入库——它们不是沉淀。

一句话判别：**这份产出的第一读者是人、且要长期留着，就是文档；被工具链吃掉才生效的，就是代码；用完即弃的，就是临时物。**

## Considered Options

- **落盘位置逐次询问用户（先问 + 推荐值）**（被否决）：interview2doc / loopspec 的旧设计。「怕写死路径污染项目结构」的顾虑已被统一根目录本身化解——`docs/jljskills/<plugin>/` 就是法定的不污染位置；逐次询问反而让产物散落，整组迁移与清理无从下手。
- **代码也收进 `docs/jljskills/`**（被否决）：代码的消费者是项目工具链（构建、import 路径、hook 执行位），挪进文档根即失效；figma2web 的 `registry.json` 已有先例裁定。
- **不立统一判据、按插件逐个约定**（被否决）：判据缺位导致每次收敛都重新讨论一轮（figma2web → figma-optimize → codeflow → support 四轮重复），本 ADR 一次钉死，后续新 skill 直接照判据落。
