# figma-optimize 需求文档

> 日期:2026-07-02 · 状态:草稿

## 1. 概述 / 背景

figma-optimize 是一个**全新的、独立的 Claude Code 插件**,面向**设计师**,帮助他们在把设计稿交付给开发之前,按一套规则与标准优化 Figma 设计图本身。

它的思路脱胎于 figma2web 插件里的 `optimize` skill,但**与 figma2web 完全不相干**:不依赖、不引用、不读写 figma2web 的任何产物(尤其是 `config.md`),自带一份独立的设计评审清单。figma2web 原样保留它自己的 `optimize`,两者各自演化。

## 2. 要解决的问题

普通设计师是「设计思维」主导,产出的界面偏审美,常常忽略结构、标准、可实现性这类「开发思维」里的东西。结果是:设计稿交付给开发后**难对接、难实现**(例如该用 auto-layout 的地方靠绝对定位堆叠、颜色游离于色板之外、图层命名无语义、近似值不统一等)。

figma-optimize 的价值,是让 **AI 补上设计师缺的那套「开发者思维」**:以「开发能不能顺畅实现」为尺度审查设计稿,并帮设计师改到位,从而**弥合设计师 ↔ 开发者之间的交付鸿沟**。「对齐开发标准」不是目的本身,而是让交付顺滑的手段。

## 3. 目标(Goals)

- 让设计师在**交付前**就把稿子优化到「利于开发落地成代码」的状态。
- 由 **AI 充当规则审计员 / 守门人**,自动揪出可验证的规则、一致性、可实现性问题,补齐设计师缺失的开发视角。
- 评审既依据**插件内置的通用设计评审清单**,也在运行时结合**设计师项目里已有的设计系统标准**。
- 采纳的改动可**由 AI 直接写回 Figma**,把机械化苦力活从设计师身上卸下。
- 产出一份可交给开发的**交付就绪报告**,作为对接凭证。

## 4. 非目标(Non-Goals)

- **不生成代码**,不做 design→code(那是 figma2web 一线的职责)。
- **不与 figma2web 耦合**:不读写它的 `config.md`,不复用它的文件,不作为它流程里的一环。
- **不替设计师做审美裁决**:AI 不主导审美方向,审美决策权归设计师;AI 只揪可验证的问题。
- **不是 Figma 设计创作工具**:只审查既有稿并按用户裁定做优化,不无中生有地设计新界面。
- **首版不引入额外 skill**(如独立的「建立标准」skill):项目标准靠运行时从 Figma 抽取,而非另立 config 产物。
- **不强制字体类改动**:云端 Figma 受限时优雅降级为「建议」。

## 5. 目标用户与使用场景

- **用户**:设计师(Figma 文件的主人、动手专家),但通常缺乏开发思维。
- **场景**:设计稿基本完成、准备交付给开发之前,设计师运行本插件做「交付前自查 + 标准化」,把不利于开发实现的地方修正掉,并带一份自查报告交接给开发。

## 6. 核心需求 / 功能清单

| 优先级 | 需求 | 说明 |
| --- | --- | --- |
| P0 | 前置校验 | figma-mcp 可用(需读取,且采纳项需经 `use_figma` 写回);拿到待审的 Figma 链接 / 范围。 |
| P0 | 内置评审清单 | 插件自带一份独立的设计评审清单(核心知识资产),以「利于开发落地」为导向,结构类与切图友好类优先级最高。 |
| P0 | 运行时装载项目标准 | 现场从 Figma 的 variables/styles 抽取色板 / 字体 / 间距 token 等当项目标准;没有则退回内置通用清单,缺口对话补。零配置、不引入 config 文件。 |
| P0 | 逐条裁定(HARD GATE) | 建议逐区块、逐条呈交,用户选 采纳 / 跳过 / 再调;写入 Figma 前必须确认,未确认不得改动。 |
| P0 | 采纳项二次选择 | 每条采纳的问题,用户再选「我自改」还是「让 AI 改」。 |
| P1 | 逐区块提取与评审 | 对每区块 `get_metadata` + `get_screenshot` + `get_design_context`;数值类靠 `use_figma` 读 x/y/w/h 审计,视觉类靠截图。 |
| P1 | 带严重度的建议列表 | 每条含:问题 / 改法 / 依据维度(清单编号) / 严重度(高/中/低)。 |
| P1 | AI 写回 Figma | 「让 AI 改」的项:先走 `figma-use` skill,再 `use_figma` 写入,每处改完 `get_screenshot` 校验。 |
| P1 | 「我自改」交付说明 | 「我自改」的项:AI 给出明确、可照做的改法说明,交设计师在 Figma 里自行修改。 |
| P1 | 复审 | 复审数值类项是否达标(理想为审计列表归零或仅剩有意保留的例外)。 |
| P1 | 交付就绪报告(可选落盘) | 产出一份报告:发现的问题、处置结果(已改 / 跳过 / 待改)、保留的例外。可当递给开发的自查单与交接凭证。 |
| P2 | 字体受限降级 | 云端 Figma 字体类改动常无法执行 → 降级为「建议」,报告中标注「受限」。 |
| P2 | snap 网格例外判断 | 对 hug 宽度、1px 描边、组件实例、矢量锚点等合理非网格值不误报。 |

## 7. 约束与依赖

- **figma-mcp(必需)**:读取 Figma;采纳项经 `use_figma` 写回 Figma。
- **figma-use skill(写回前强制前置)**:调用 `use_figma` 之前必须先走 figma-use skill。
- **不依赖 superpowers**(无「计划→执行」的编码环节)。
- **不依赖、不触碰 figma2web**。
- 云端 Figma 执行上下文受限,字体类改动常无法完成 → 降级为建议,不强制。

## 8. 成功标准

- 数值类审计项(off-grid 间距、游离色、近似值不统一等)复审后**归零**,或仅剩用户有意保留、并在报告中标注的例外。
- 结构类、切图友好类的高严重度问题均已处置(改到位或明确跳过并记录)。
- 设计师拿到「交付就绪报告」后,可直接把稿子交付开发,**开发对接与实现的摩擦显著降低**。

## 9. 开放问题(待确认)

- **插件物理位置 / 仓库**:由用户自行确定,需求文档中不写死路径。
- **调用入口命名**:暂定 `figma-optimize:optimize`(单 skill 插件),具体名后续微调。
- **交付就绪报告的落盘位置约定**:运行时向用户询问,还是默认某约定目录(如 `docs/figma-optimize/`)?待定。
- **交付就绪报告的模板结构**:具体字段与格式待细化。
- **内置清单相对 figma2web 版是否需要增删**:当前拟复用一份副本、保留「利于 coding」的优先级排序;是否针对设计师场景做裁剪待定。
