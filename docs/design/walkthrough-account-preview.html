Overview

The code that turns a Firebase identity into a backend account and governs that account's whole lifecycle. Firebase owns authentication; the backend owns an account mirror — one users row per Firebase identity, keyed by firebase_uid. Almost every request is "verify the token → find (or provision) the matching row → act."

The lifecycle — a small state machine over is_active + deactivated_at:

none— provision → active— POST /deactivate → deactivated— ≥7d + re-register → hard-deleted · email freed

1. The data model — User + UserProfile

The split is deliberate: User holds identity + lifecycle (the immutable firebase_uid, unique email, and the timestamps is_active / last_login / deactivated_at); UserProfile holds mutable settings + the onboarding flag. Every child table cascades off User — which is what lets a hard-delete clean up everything downstream.

2. Verifying the token — verify_firebase_token

Parses the Authorization: Bearer header → decoded token, 401-ing on anything malformed. Before real verification there's a dev-only shortcut: with debug_mode on and a DEBUG_BYPASS_TOKEN prefix, the JSON payload is used as the token — the frontend-testing mechanism. Loudly logged, inert in production.

The real path calls auth.verify_id_token() (signature + exp/iat/aud/iss/sub). Each failure maps to a specific 401 — expired / revoked / invalid are distinguished so the client can react.

3. Token → User row: the two-tier dependency

_resolve_authenticated_user looks up the row by firebase_uid and enforces three invariants: no row → 404 (deliberately not auto-creating), inactive → 403, email mismatch → 403.

⚑ War story — why async wraps a sync helper

A sync yield-dep around user_id_scope blew up in production with "Token was created in a different Context". FastAPI runs __enter__/__exit__ of a sync yield-dep in two different copy_context() threadpool calls, so the contextvar set() couldn't be reset(). Making the wrapper async keeps set + reset in one event-loop Context; the blocking work stays in the sync inner dep.

4. Signup — POST /user

Called right after Firebase signup. Asserts token uid matches the body (403 otherwise), then is idempotent: an existing row is returned as-is (no mutation, no log). Only a new identity falls through to provision_user, wrapped in explicit commit / rollback.

5. The shared create path — provision_user

Guards the email unique constraint. If the email is taken, it proceeds only when that account is stale-deactivated past its grace window (hard-delete it, free the email) — otherwise 409 rather than hijack. On success: users row + default UserProfile + linked family invites.

6. Login — POST /login and the self-heal

If signup-time sync never landed, you'd have a Firebase identity that can authenticate but has no backend row — it could never log in. Login closes the gap: no row → self-provision from the token's email via the same provision_user. Deactivated still 403s; success stamps last_login.

7. Deactivate — POST /deactivate

Flips is_active=False + stamps deactivated_at=now (row kept for audit), cleans up family data, and deletes the Firebase identity. Two resilience choices: the Firebase delete is idempotent (UserNotFoundError is fine), and other Firebase errors are logged but non-fatal.

8. The grace window — hard-delete eligibility

is_account_hard_delete_eligible is a pure predicate: inactive and deactivated_at ≥ 7 days old. hard_delete_stale_deactivated_user then physically db.delete(user) (cascading away every child row), freeing the email — the piece provision_user calls during re-registration.

9. In-place onboarding reset — reset_user_onboarding

Backs the CLI (scripts/db_user_admin.py reset-user) and the DEBUG-only endpoint. Deletes every per-user row the onboarding flow creates — parents first so DB-level ON DELETE CASCADE reaps children — then flips setup_completed back to False. The users row + Firebase identity are kept, so the same login replays onboarding from a clean slate.

Recap — one account, cradle to grave

Firebase id→ provision → active→ deactivate → deactivated→ <7d: login 403 → ≥7d + re-register →hard-deleted
  • Firebase authenticates; the backend mirrors. One users row per firebase_uid; reads 404 rather than auto-create; provisioning funnels through one provision_user.
  • Deactivation is soft + delayed. is_active + a 7-day window keeps the row auditable and lets the same email return; the Firebase identity dies immediately.
  • Messy realities handled explicitly. Idempotent Firebase deletes, non-fatal cleanup failures, the async/sync contextvar split, the 409-vs-hard-delete fork.
